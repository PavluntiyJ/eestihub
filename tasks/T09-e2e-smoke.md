# T09 — Playwright e2e-смоки

**Прочитай сначала:** `docs/CONTEXT.md` (§4, §6, §7).
**Зависимости:** T07 принята. **Роль:** Frontend/QA (TS).
**Ветка:** `feat/t09-e2e` от текущего `master`; перед сдачей —
`git rebase master` (параллельно в `master` идёт backend-задача T08,
пересечений по файлам нет). Мерж после ревью делает оркестратор.

## Цель

Браузерные смоук-тесты ключевых пользовательских сценариев. До сих пор
всё проверялось curl'ом по SSR-HTML; клики, гидрация и клиентский fetch
формы не покрыты ничем. После T09 каждое ревью гоняет e2e.

## Что сделать

1. Зависимость: `@playwright/test` в devDependencies (единственная
   новая). Браузер — только chromium (`npx playwright install chromium`).
   Если в среде chromium не запускается (нехватка системных библиотек) —
   зафиксируй точную ошибку в «Заметках для оркестратора» и сдай задачу
   с этой оговоркой, не выкачивая полсистемы.
2. `frontend/playwright.config.ts`: `testDir: './e2e'`, `baseURL`
   `http://localhost:3000`, `webServer` — прод-фронтенд
   (`npm run build && npm run start`, `reuseExistingServer: true`).
   Бэкенд тесты НЕ поднимают: перед запуском он должен работать на :8000
   (uvicorn из `backend/`, venv) — задокументируй это в комментарии
   конфига. Скрипт в `package.json`: `"e2e": "playwright test"`.
3. Тесты в `frontend/e2e/` (TS strict, без `any`), сценарии:
   - **redirect+home**: `/` → `/en`, h1 виден, бейдж API online.
   - **язык**: на `/en` клик по `ET` → URL `/et`, h1 сменился на
     эстонский, `<html lang="et">`.
   - **навигация**: с `/en` клик «Calculator» → `/en/calculator`,
     `aria-current` переехал; клик «Rent» → `/en/housing`.
   - **калькулятор (главный сценарий)**: на `/en/calculator` ввести
     3000, выбрать 2%, submit → появляются 4 карточки режимов,
     на странице виден net Tööleping `€2,409.76` и бейдж «best»
     у лучшего режима.
   - **housing**: на `/en/housing` таблица содержит 8 строк районов
     (Kesklinn … Pirita), график отрисован (canvas/svg присутствует).
   - **невалидный ввод**: очистить поле дохода → submit заблокирован,
     видно сообщение об ошибке валидации.
   Локаторы — по ролям/aria/тексту (`getByRole`, `getByLabel`),
   не по CSS-классам.
4. Прогнать всю сьюту локально с работающим бэкендом; приложить итог
   (`N passed`) в журнал.
5. Коммит: `test(frontend): playwright e2e smoke suite`.

## Чего НЕ делать

- Не менять прод-код (страницы, компоненты, словари, api-клиент). Если
  для локатора не хватает aria/роли — запиши в «Заметки для
  оркестратора», не патчи компоненты сам.
- Не трогать backend и `docker-compose.yml`.
- Не добавлять другие тест-фреймворки, скриншотные сравнения, CI-конфиги
  (CI — отдельная задача следующей итерации).

## Критерии приёмки

- [ ] `npm run e2e` с поднятым бэкендом: все сценарии зелёные.
- [ ] Сценарий калькулятора реально кликает submit и проверяет цифры
      ответа (не только наличие формы).
- [ ] `npm run build` по-прежнему чистый; прод-код не изменён
      (диф только e2e/, конфиг, package.json, lock).
- [ ] Локаторы устойчивые (роли/aria/текст), без CSS-классов.
- [ ] Ветка перебазирована на актуальный `master`.
